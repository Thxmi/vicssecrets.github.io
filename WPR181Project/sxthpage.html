Web Application Architecture

Web Application Architecture is described as the interaction between different applications, databases, and middleware on the web. It ensures that the communication occurs simultaneously. A web application has to include all subcomponents with the addition of external applications interchanges for the entire software application. This is used to combat the code parsed by the web browser. The code might not have specific instructions to tell the browser how to deal with different user input. There are to different codes which run simultaneously with each other. Client-side-code which is the code that runs in the browser and responds to some user input. Server-side-code which is code that runs in the server and responds to HTTP requests. The type of code run on the server will be decided in respects to the web browser. Code that can respond to HTTP can run on a server. The sever-side-code is responsible for creating the page requested and stores different type of data. It is never seen by the user. The Client-side-code uses different codes to be seen and used by the user. While reacting to user input.

Web applications have four components :

UI/UX Web Application Components
These are a part of the layout plan of a web application. They do not operate the web application architecture.
Structural Components
The two dominate structures of a web application are client and server sides.
Client Component
This is developed in HTML, CSS, and JavaScript. The component is run on the client web browser. It is the component that the end-user interacts with.
Server Component
This can be one or more build that uses different programming languages and frameworks. It has at least two parts: app logic and database. The former controls the web application while the latter is where all persistent data is stored.

Depending on the number of databases and servers available a model can be created. They can include:

One Web Server, One Database
this model is the simplest but the most unreliable conversely. If One of the components, the server, goes down, the entirety of the website will go down.
Multiple Web Servers, One Database
The idea behind this model was that a server does not store information. So, the servers can process information parallel to each other and store it in the database. If one of the servers go down the others will take precedence. This increases its reliability compared to the first one. You need at least two servers to form this model.
Multiple Web Server, Multiple Database-
This is the most efficient manner as there is no single point of weakness in the model. You could store the same data and distribute them evenly across the database, not more than 2 databases are required. The other method would include storing the data in the different databases, this method although is susceptible to data loss or corruption.

The pattern of interaction between various web application components can be described as web applications architecture. The type of web application architecture depends on how the app logic is distributed among the client and server sides.

There are mainly three types of web application architecture:

Single-Page Applications
A single page web applications allow for dynamic interaction by means of providing updated content to the current page. This is used to avoid loading completely new pages from the server each time for a user action.
Microservices-
These services execute a single functionality. This allows the framework to improve productivity and increase the entire deployment process speed.
Serverless Architecture
An application developer would contact a third-party cloud server holder for outsourcing server with infrastructure management. This would be beneficial to the application for executing code without bothering with infrastructure related task.
